## 1. 项目边界

### 1.1 是什么

- ✅ **Agent 优先**的记忆后端
- ✅ **Markdown 作为 source of truth**
- ✅ **混合搜索**（Vector + BM25）
- ✅ **分层存储**（热/短期/长期）
- ✅ **兼容 OpenClaw 和 ZeroClaw**
- ✅ **人类可读可调试**

### 1.2 不是什么

- ❌ **不是数据库项目**：不追求极致存储性能
- ❌ **不是搜索引擎项目**：不提供复杂查询语言
- ❌ **不是通用向量数据库**：专注于 Agent 记忆场景
- ❌ **不是分布式系统**：单机部署优先

---

## 2. 硬边界（Hard Boundaries）

### 2.1 数据完整性

- **禁止**：丢失用户数据
- **禁止**：破坏 Markdown 文件格式
- **禁止**：无法从 Markdown 重建索引

**保证**：
- 所有数据都有 Markdown 备份
- 向量索引可以随时从 Markdown 重建
- 提供数据迁移和回滚工具

### 2.2 测试质量

- **禁止**：降低测试覆盖率要求（> 80%）
- **禁止**：使用 `#[ignore]` 跳过测试
- **禁止**：合并未通过测试的代码
- **禁止**：绕过代码审查流程

**保证**：
- 所有测试必须全绿通过
- CI/CD 自动运行测试
- 测试失败阻断合并

### 2.3 可调试性

- **禁止**：黑盒设计（无法调试）
- **禁止**：隐藏关键数据（如 embedding 向量）
- **禁止**：不可逆操作（如自动删除数据）

**保证**：
- 提供详细的日志输出
- 提供诊断工具（`hipocampo status`）
- 所有操作都有对应的查询命令

### 2.4 性能底线

- **搜索延迟**：< 100ms（P95）
- **索引吞吐量**：> 1000 chunks/s
- **内存占用**：< 100MB（热记忆）

**保证**：
- 定期性能测试
- 性能回归必须有文档说明
- 提供性能优化指南

## 3. 禁止项（Prohibitions）

### 3.1 架构禁止

- ❌ **禁止**：将数据库作为 source of truth
- ❌ **禁止**：实现复杂的查询语言（如 SQL）
- ❌ **禁止**：引入重量级依赖（如 TensorFlow）
- ❌ **禁止**：使用不可移植的存储格式

### 3.2 实现禁止

- ❌ **禁止**：`unwrap()` 在生产代码中（必须处理错误）
- ❌ **禁止**：`panic!()` 作为正常流程
- ❌ **禁止**：硬编码配置（如 API keys）
- ❌ **禁止**：忽略错误处理

### 3.3 流程禁止

- ❌ **禁止**：直接提交到 main 分支（必须 PR）
- ❌ **禁止**：跳过代码审查
- ❌ **禁止**：合并未测试的代码
- ❌ **禁止**：删除或修改测试用例以"通过"测试

### 3.4 文档禁止

- ❌ **禁止**：缺少公共 API 文档
- ❌ **禁止**：过时的文档（与代码不符）
- ❌ **禁止**：缺少示例代码

## 5. 风险控制

### 5.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| sqlite-vec 不可用 | 高 | Fallback to in-process cosine similarity |
| Embedding API 限流 | 中 | 本地缓存 + 降级到关键词搜索 |
| 数据迁移失败 | 高 | 备份机制 + 回滚工具 |

### 5.2 项目风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 需求理解偏差 | 高 | 早期澄清 + 原型验证 |
| 进度延误 | 中 | 里程碑检查 + 调整计划 |
| 技术债务累积 | 中 | 定期重构 + 代码审查 |

## 7. 应急预案

### 7.1 数据损坏

1. 停止服务
2. 从 Markdown 重建索引
3. 验证数据完整性
4. 恢复服务

### 7.2 性能问题

1. 诊断（`hipocampo status --verbose`）
2. 分析日志
3. 优化索引
4. 必要时回滚版本

### 7.3 Bug 紧急修复

1. 创建 hotfix 分支
2. 修复 + 测试
3. 快速审查
4. 发布 patch 版本
